# 工具设计规范检查器 (Tool Spec Checker)

当你被要求评估、编写、修改或审查一个工具（Tool/Function）的定义、声明或 Schema 时，你必须严格按照以下五大核心规范进行对照检查，并在回复中给出结构化的反馈：

## 1. 职责清晰 (Clear Responsibility)
* **要求**：这个工具到底是做什么的，其核心边界必须清晰。
* **检查重点**：
    * 工具是否保持单一职责？（拒绝多功能组合的“万能工具”）
    * 工具的描述（description）是否准确传达了其核心用途和适用边界？

## 2. 输入清晰 (Clear Input)
* **要求**：参数结构应该明确、稳定、可校验，最好是结构化字段，而不是一大段模糊的字符串。
* **检查重点**：
    * 参数是否声明了具体的类型（如 `string`, `number`, `boolean`, `array`, `object`）？
    * 是否合理配置了 `enum`（枚举值）或 `required`（必填字段）限制？
    * 尽量避免让模型输入大段非结构化的文本，除非该工具本身就是纯文本分析器。

## 3. 输出清晰 (Clear Output)
* **要求**：输出要求稳定，方便模型和程序后续处理。
* **检查重点**：
    * 输出格式是否可预测（推荐返回结构化的 JSON 或易于解析的对象）？
    * 输出中是否包含了后续步骤所需的所有关键标识（如 ID、状态码、资源状态等）？

## 4. 权限清晰 (Clear Permissions)
* **要求**：哪些可以做，哪些坚决不能做。对于不清楚能不能做的，要请求人类干预。
* **检查重点**：
    * 该工具是否涉及高风险、写操作或敏感数据访问？
    * 当操作可能带来不可逆风险或处于边界模糊的地带时，工具是否具备确认机制，或模型是否被明确要求在调用前执行 `Ask human for intervention`（请求人类干预）？

## 5. 错误可处理 (Handleable Errors)
* **要求**：工具失败时要能区分不同的错误类型，以便模型和系统进行针对性修复、重试或安全退出。
* **检查重点**：
    * 返回值或异常中是否能清晰区分以下错误类型：
        * **参数错误** (Parameter Error) -> 模型可尝试修正参数后重新调用。
        * **权限错误** (Permission Error) -> 模型必须停止尝试并如实告知用户。
        * **网络错误** (Network Error) -> 模型可提示用户或进行有限次重试。
        * **数据为空** (Empty Data) -> 属于正常流程，但需明确返回空状态而非抛出未知异常。

## 执行指令

当触发此技能或用户显式输入 `/check-tool-spec` 时，请执行以下工作流：
1. **逐条审计**：对照上述 5 点规范，详细指出当前工具设计中“符合”与“不符合”的具体表现。
2. **风险提示**：如果存在权限模糊或错误分类不清的情况，重点加粗指出。
3. **重构建议**：提供优化后的具体修改建议，并直接输出修改后的完整工具 Schema 定义（例如符合 Anthropic Tool standard 的 JSON Schema 或 TypeScript 类型定义）。
