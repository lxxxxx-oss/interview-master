# 自动化开发日志、测试与 Bug 修复文档记录规范

为了保持项目的可维护性，在完成任何阶段性的关键操作（特别是代码变更、测试或 Debug）后，你必须执行以下文档记录流程。

## 1. 触发标准（含测试与 Bug 修改）
* **新功能落地**：完整实现了某个业务逻辑（如 Agent 核心逻辑、MCP 节点对接等）。
* **架构与配置变更**：修改了系统核心配置或数据库 Schema。
* **Bug 修复与重构**：定位并修复了复杂的系统级或业务级 Bug，调整了核心代码结构。
* **测试用例编写与执行**：编写了新的测试脚本、执行了自动化测试或记录了测试反馈。
* **用户明确指示**：用户表示阶段性任务完成或要求记录文档。

## 2. 严格的子目录记录位置（与项目实际目录完全一致）
你必须严格按照当前项目的 `docs/` 目录及其子文件夹结构进行更新，严禁在根目录乱建文件。请根据操作类型精准分类：

### 📁 docs/progress/ （进度与 Bug 修复子目录）
* **日常进度日志**：将详细的开发变更、时间戳和后续计划，追加到该目录下的日常进度文件中。
* **Bug 修复记录**：如果是修复了 Bug，必须在该目录下的日志中详细记录：**Bug 表现、根本原因分析（Root Cause）、修复方案以及防御性重构措施**。

### 📁 docs/tests/ （测试专属子目录）
* **测试文档更新**：如果是编写了新测试、执行了接口测试、或进行了功能验证，必须在 `docs/tests/` 目录下创建或更新对应的测试说明文档（如测试用例、测试报告、覆盖率说明等）。

### 📄 docs/ 根目录文件
* **项目整体状态 (`docs/IMPLEMENTATION_STATUS.md`)**：如果推进了宏观进度或修复了里程碑式的 Bug，同步更新此文件，标记已完成或已修复的事项。
* **接口与工具变更 (`docs/API.md`)**：如果新增/修改了函数接口、Agent 工具定义（Tool Schema）或后端 API，同步更新此文件。
* **核心技术记录 (`docs/TECHNICAL_HIGHLIGHTS.md`)**：如果攻克了高难度的技术死角、优化了算法性能或完成了重大的架构设计，将思路追加至此。

## 3. 日志记录结构 (针对 progress 和 tests 目录)
写入子目录的记录必须包含：
* **[时间戳]**：当前的日期与时间。
* **操作类型**：[功能新增 / 代码重构 / Bug修复 / 测试执行 / 配置变更]
* **变更概述**：简练说明做了什么、修复了什么问题，或测试了什么功能。
* **影响范围/测试结果**：受影响的文件，或者测试通过率/发现的遗留问题。
* **Next Steps**：下一步待办或优化方向。

## 执行指令

当你检测到上述任务完成并触发此技能时，请直接执行以下步骤，无需再次询问人类：
1. **自动盘点**：检查本次对话中你实际修改了哪些代码、修复了哪些 Bug 或编写了哪些测试。
2. **精准分流**：根据第 2 点的路径要求，分别进入 `docs/progress/` 或 `docs/tests/` 等对应子目录更新 Markdown 文件。
3. **静默确认**：保存文件后，向用户简要汇报：“*已将本次变更（含测试/Bug修复）同步至 docs/ 对应子目录下的相关文档。*”
